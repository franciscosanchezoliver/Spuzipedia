Modelo de prototipos

Definición

Los modelos de prototipo son modelos de desarrollo evolutivo. El prototipo se debe de crear en muy poquito tiempo (para ello tenemos que tener las herramientas adecuadas) ya que el desarrollo del prototipo no debe de tomarnos mucho tiempo.

Hay que hacer un diseño rápido de los aspectos del software que serán visibles para el cliente. Este diseño será el prototipo. Este prototipo debe ser presentado y probado por el cliente el cual nos dará un feedback. Gracias a este prototipo podemos descubrir nuevos requisitos o refinar los que ya había.

La idea es ir refinando el prototipo para que el prototipo se ajuste a las necesidades del cliente. Gracias al prototipo podemos ver que es lo que quiere el cliente y que además el cliente vea resultados rápidamente.

Etapas

  1. Comunicación con el cliente.
  2. Modelado (diseño) del prototipo.
  3. Construcción del prototipo.
  4. Desarrollo del prototipo.
  5. Entrega y retroalimentación del cliente.
  6. Entrega del desarrollo final.

Ventajas

  • El modelo de prototipos es útil cuando el cliente comprende los objetivos generales del software pero no comprende los requisitos de entrada/salida.
  • Cuando el desarrollador no está seguro de si se está haciendo correctamente la interacción humano-máquina.
  • Podemos reutilizar código que hemos usado para la creación del prototipo.
  • Ayuda tanto al desarrollador de software como al cliente a entender cual será el resultado final una vez que los requisitos estén satisfechos
    • Conseguimos que el cliente se involucre más en el desarrollo del software.

Inconvenientes

  • Debido a querer crear un prototipo lo más rápido posible, se suele desatender la calidad del software y no pensar en su futuro mantenimiento. Esto obligaría a tener que reconstruir el prototipo desde 0 cuando empezamos a hacer programa, ya que como hemos intentado hacer el prototipo tan rápido el código de dicho prototipo puede que no sea reutilizable para empezar a hacer el programa real. Sin embargo es muy posible que el cliente no quiera que empecemos a construir el sistema desde 0. El lo que querrá será que construyamos el sistema partiendo del prototipo, esto lo convertiría en un prototipo evolutivo pero partiendo de un estado poco recomendable ya que hicimos el prototipo rápido y corriendo y su código será bastante malo.
  • Por tal de hacer un prototipo rápido, el desarrollador puede tomar decisiones erróneas como por ejemplo elegir el lenguaje de programación que nos permita un desarrollo más rápido pero que a la larga esto sea peor.

Conclusiones

Aunque haya ciertos inconvenientes, la construcción de prototipos suele ser un paradigma efectivo en la ingeniería del software. Pero hay que definir unas reglas de juego entre el cliente y el desarrollador:

  • El prototipo construido es una forma de definir requisitos de entrada y salida.
  • El prototipo debe descartarse (al menos parte de él).
  • Una vez que se comience a desarrollar el sistema real hay que centrarse en la calidad del sistema y no tanto en la velocidad de desarrollo.
