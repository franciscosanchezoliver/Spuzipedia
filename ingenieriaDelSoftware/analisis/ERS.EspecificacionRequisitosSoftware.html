Especificación de Requisitos del Software (ERS)

Definición

Describe el comportamiento del sistema. Debe tener un conjunto de casos de uso que describen las interacciones con el usuario. Los casos de uso representan los requisitos funcionales del sistema. También tiene que estar reflejados los requisitos no funcionales en la ERS, estos imponen restricciones en el diseño o en la implementación.
La ERS va enfocada tanto al equipo de desarrollo como al cliente. Para que sea comprensible por todas las partes se debe utilizar un lenguaje informal.

Características de una buena ERS

Estás características se describen en el estándar IEEE 830-1998.

  • Completa: deben de estar todos los requisitos.
  • Consistente: deben de ser coherentes con los propios requisitos y con los demás documentos de la especificación.
  • Inequívoca: los requerimientos no se pueden malinterpretar.
  • Correcta: el software debe cumplir con los requisitos de la especificación
  • Trazable: se podrá consultar la historia, ubicación o aplicación de un ítem.
  • Priorizable: se podrá hacer una jerarquia con los requisitos según su importancia para el negocio, dándole varias importancias: esenciales, condicionales y opcionales.
  • Modificable: aunque todo requisito puede ser modificable, debe ser fácilmente modificable.
  • Verificable: debe existir un método finito sin costo para poder probarlo.

Tipos de requisitos

  • Requisitos de usuarios: necesidades que los usuarios expresan verbalmente.
  • Requisitos del sistema: los componentes que necesita el sistema para poder hacer diferentes tareas.
  • Requisitos funcionales: servicios que debe de proporcionar el sistema
